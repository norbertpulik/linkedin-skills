# SDET Content Skill

## Audience

Primary: SDETs, QA leads, and automation engineers — practitioners who know the tools and will immediately notice vague claims.

Secondary: engineering managers and tech leads evaluating test-automation maturity — they care about outcomes and risk, not tool names in isolation.

Write for the primary audience. The secondary audience reads posts that are precise and credible; they won't engage with posts that sound like a consultant pitch.

## Positioning

Senior SDET, 6 years. Playwright/TypeScript as the primary automation stack. CI/CD via Jenkins and Docker. Domain experience across SCADA/industrial systems and fintech/banking — two environments where test failures have real consequences (safety or regulatory, not just user complaints).

Credibility beats virality. An engineer audience trusts specifics over superlatives. Lead with what actually happened in a real pipeline or test suite.

## Content pillars

### P1 — Playwright in production

What running Playwright at scale actually looks like, beyond the docs.

Covers: fixture architecture for shared state, `page.route()` for network interception vs. mock-layer traps, trace viewer as a debugging artifact (not just a CI log), parallelization and worker isolation, retry strategies that don't hide flakiness, TypeScript type safety in test code.

Angle: "here's what the docs don't cover once you have 500+ specs."

### P2 — CI/CD quality gates that mean something

The gap between "we have tests in CI" and "CI actually catches regressions."

Covers: test selection (run the right tests on the right trigger, not all tests always), flake detection and quarantine policies, coverage gates anchored to specific risk areas not global %, failure attribution (was it the code or the environment), Docker test environment drift, Jenkins pipeline design for test stability.

Angle: "a passing CI build is not evidence of quality — it's evidence that your tests ran."

### P3 — Testing in regulated domains

What changes when the system under test is a SCADA controller, a banking transaction engine, or anything with an audit trail.

Covers: determinism requirements that make flaky tests unacceptable (not just annoying), test environment isolation near safety-critical or compliance-boundary systems, audit evidence generated by tests (not just coverage reports), approach to testing state machines with real-world timing constraints, fintech-specific: idempotency testing, contract testing for payment APIs.

Angle: practitioner observations from two domains most QA content ignores.

### P4 — AI tools in the QA workflow

Honest, tool-specific takes — where AI helps and where it produces confident garbage.

Covers: Copilot/AI for test scaffolding (good for boilerplate, dangerous for assertions), AI-generated tests that pass by testing the mock not the code, coverage theater from AI-generated suites, where LLMs are actually useful in testing (test data generation, assertion suggestion, parsing logs), AI-assisted test maintenance for selector drift.

Angle: "used Copilot to generate 80 unit tests last week. 60 passed. 12 were testing the mock, not the code." — concrete, reproducible, not a vendor pitch.

### P5 — QA strategy and engineering influence

How to make the case for test infrastructure investment to people who control the roadmap.

Covers: framing flaky CI as an engineering productivity cost (not a QA problem), building the business case for e2e investment using prod incident data, what "shift-left" actually requires (test infrastructure, not just attitude), how to run a test strategy review that engineering managers will act on, the conversation between SDET and EM when test debt reaches a tipping point.

Angle: "here's the framing that got budget approved" — outcomes and risk language, not testing-theory language.

## Lead magnet

**Playwright Production Checklist** — a reference for running Playwright in real CI environments. Reference it when natural (e.g., in posts about fixture design, network interception, or CI integration). Never force it. Never end a post with "DM me for the checklist."

Natural placement: "the full checklist is in my featured section" or "I cover this in the Playwright Production Checklist I put together" — one mention max per post.

## Drafting rules

1. **Lead with a real, specific problem from CI or Playwright work.** Not "testing is hard" but "our Jenkins pipeline was timing out on 14% of runs and we couldn't reproduce it locally."
2. **One concrete artifact per post**: a code snippet, a number from a pipeline run, a before/after config, a specific error message. If the post has none of these, it needs to be rewritten before it can be approved.
3. **No marketing vocabulary.** No rule-of-three filler. No "this changed everything." See `../../references/voice-rules.md` for the full blacklist.
4. **No growth-hack hooks that overpromise.** "I 10x'd our test speed" requires the actual baseline and result. Without both numbers, reframe to state what changed specifically.
5. **Precision is the hook for this audience.** "Playwright's `page.route()` intercepts at the network layer, not the mock layer" will outperform "Playwright is a game-changer" with SDETs every time.
6. **Column the pillar.** Tag each draft with which pillar it serves (P1–P5) so content across sessions doesn't cluster on one topic.
7. For posts: 900–1,300 chars, hook in first 210 chars, 0–2 hashtags at end, no external links in body (move to first comment).
8. For comments: 200–350 chars, one concrete detail not already in the parent post, end with a practitioner-specific question.

## Output format

For every draft, provide:

1. **Pillar** (P1–P5) and **format** (blunt observation / before-after / number-drop / counterintuitive claim / tool-specific observation)
2. **Draft text** (full, copy-paste ready)
3. **Artifact check**: what is the concrete artifact in this draft? If none, flag it.
4. **Hook check**: first sentence. Does it state a specific problem, number, or observation? If it starts with a claim, flag it.
5. **Char count** and approval prompt
